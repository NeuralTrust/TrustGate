# Proposal: RUN-281 New data model + admin CRUD (control plane)

## Intent
Implement AgentGateway's new control-plane data model and its admin CRUD
surface: domain aggregates, Postgres schema + in-code migrations,
`pgx/v5` repositories, application use cases, and Fiber admin HTTP
handlers for `Gateway`, `Backend(LLM|A2A|MCP)`, `Consumer`, `Policy`,
`Auth`, plus the three many-to-many join tables. The shape is inspired by
TrustGate but **not** a 1:1 port — the new model collapses the old
`gateway/service/upstream/rule/consumer/plugin-chain` topology into a
tighter `Gateway / Backend / Consumer / Policy / Auth` graph (epic
RUN-279).

## Scope
### In Scope
- RUN-295: domain aggregates + relationships + Postgres schema + in-code
  migrations (`pkg/domain/{gateway,backend,consumer,policy,auth}`,
  `pkg/infra/database/migrations/<unix_ts>_*.go`).
- RUN-296: per-aggregate repositories on `pgx/v5` + `database.WithTx`
  (`pkg/infra/repository/<entity>/repository.go`), including the diff
  logic that maintains the three join tables consistently.
- RUN-297: app services per entity (`pkg/app/<entity>/{creator,updater,
  deleter,finder}.go`) following AGENT.md §10.2 — interface + impl +
  `//go:generate mockery` in one file, mocks in `mocks/`.
- RUN-298: admin HTTP handlers (one action per file in
  `pkg/api/handler/http/`), request DTOs in
  `pkg/api/handler/http/request/`, response DTOs in
  `pkg/api/handler/http/response/`, minimal helpers
  (`pkg/api/handler/http/helpers/`), and the new admin routes wired
  through `pkg/server/router/admin_router.go`.
- DI fill-in for the five entity modules
  (`pkg/container/modules/{gateway,backend,consumer,policy,auth}.go`,
  currently `return nil` placeholders) and `pkg/container/modules/api.go`.

### Out of Scope
- RUN-299 cache invalidation events — **deferred**; depends on B.1
  cache subsystem (RUN-291) which is paused.
- Hot-path Policy matcher / resolver (B.4 / RUN-283).
- Plugin chain wiring, plugin manager, plugin catalog (B.3 / RUN-334 +
  RUN-293). `Policy` aggregate persists no plugin-chain field in B.2.
- `Auth` type implementations (B.7); B.2 persists a generic descriptor
  (`type`, `config`) with no verification logic.
- Audit log integration (B.9).
- Streaming, provider adapters, load balancing, hot-path traffic
  forwarding (B.4 / B.5).
- Read-through caching in repositories — repositories are strictly DB.
- Admin API authentication / authorization (B.7).
- Soft delete (use FK `ON DELETE CASCADE` instead).
- Multi-tenant / Team filters on listings — single-team semantics.

## Capabilities
### New Capabilities
- `domain-model`: aggregates `Gateway`, `Backend(type=LLM|A2A|MCP)`,
  `Consumer`, `Policy`, `Auth` and the three join aggregates
  `consumer_backend`, `consumer_policy`, `consumer_auth`.
- `persistence-model`: Postgres schema + in-code migrations + per-aggregate
  pgx repositories with transactional writes for many-to-many diffs.
- `app-services`: per-entity `creator/updater/deleter/finder` use cases
  with mockery-generated test doubles.
- `admin-crud-api`: REST admin endpoints `POST/GET/PUT/DELETE
  /v1/<entity>` and `GET /v1/<entity>` listing with offset pagination
  and substring-name filtering.

### Modified Capabilities
- `dependency-injection`: wire concrete repositories, app services, and
  handlers into the five entity modules; extend `modules/api.go` with the
  new admin handler providers.
- `http-server`: extend `admin_router.go` with `/v1/<entity>` routes;
  proxy router stays untouched in B.2.

## Approach
**Per-entity chained PRs** (approach D from `explore.md` §5). Each entity
ships in three chained PRs:

- `<entity>-a` — domain aggregate + migrations + pgx repository (closes
  RUN-295 and RUN-296 jointly for that entity).
- `<entity>-b` — app services + mockery mocks (closes RUN-297 for that
  entity).
- `<entity>-c` — admin handlers + request/response DTOs + router wiring
  + DI module fill-in (closes RUN-298 for that entity).

Entity order: **Gateway → Backend → Consumer → Policy → Auth**. Gateway
goes first to establish the shared shape (helpers, common errors, repo
folder layout). Consumer is last among the n↔n owners so the FK targets
(Backend, Policy, Auth) already exist on `main`.

Backend uses a single-table + JSONB-config polymorphism (`type`
discriminator). Identifiers are UUID v4 generated by the app layer.
Many-to-many associations are owned by the `Consumer` aggregate and
diffed inside `WithTx`. No soft delete; FK `ON DELETE CASCADE` on the
three join tables.

## Risks
| Risk | Mitigation |
|---|---|
| Backend JSONB validation drift (config payload doesn't match `type`). | Strict per-type validators in `pkg/app/backend/creator.go` and `updater.go`; failing fast with a domain sentinel error. |
| Consumer aggregate touches 3 join tables → PR likely > 400 LOC budget. | Split `Consumer-a` into `Consumer-a1` (entity + migrations) and `Consumer-a2` (repo + diffing logic). Other splits documented in `explore.md` §8. |
| Migration ordering — child tables must follow parent tables (gateways before backends; consumer_* joins last). | Unix-timestamp prefixed filenames; explicit ordering in the timestamps, not in lexical naming. Registration order is enforced by the manager's sort. |
| AGENT.md §11 says repo subfolder; TrustGate uses flat — contributors may copy the wrong shape. | AGENT.md is the source of truth (just updated). Propose/design enforce subfolder. Reference `TrustGate/pkg/app/<entity>` only for app-service shape, not repository layout. |
| RUN-299 (cache invalidation) deferred; repos do not call any cache client today. | Keep repository constructor signatures stable so RUN-299 can wrap them rather than rewrite them when RUN-291 lands. |
| `Auth.Type` and `Policy.Action` accepted-value lists are not yet finalised. | Use defaults from the domain glossary (`Allow/Log/Mask/Block` for Policy actions; well-known type strings for Auth). Final lists confirmed before propose closes; design records them explicitly. |
| Admin auth + audit not integrated → handlers will work without auth. | Acceptable per RUN-281 issue body (auth + audit live in B.7 / B.9 as later middleware additions). No handler refactor expected. |
| `mockery` not yet wired into the repo (B.0 didn't install it). | Tasks include a one-time addition of `mockery` to `tools.go` or Makefile target before any `<entity>-b` PR. Hand-rolled mocks are forbidden by AGENT.md §10.5. |

## Rollback Plan
Each entity slice is independently revertible. Revert order **inside an
entity** is `c → b → a` (handler routes off the router first, then app
services + mocks, then domain + migration + repo). Migrations have
explicit `Down` functions so a rollback can be applied per migration
ID. Reverting the `<entity>-a` PR removes the entire entity safely as
long as `<entity>-b` and `<entity>-c` were already reverted.

Across entities, revert order is `Auth → Policy → Consumer → Backend →
Gateway` (reverse of merge order) because Consumer's joins depend on
Backend/Policy/Auth and everything depends on Gateway helpers/common
errors.

## Open Questions Resolved at Propose
- §6.1 — Backend polymorphism: **single-table + JSONB**.
- §6.2 — Identifiers: **UUID v4, app-generated**.
- §6.3 — Soft delete: **none in B.2**.
- §6.4 — Many-to-many writes: **Consumer aggregate owns them; diffed
  inside `WithTx`**.
- §6.5 — `Auth.Type` accepted strings: default to the well-known set
  from the domain glossary; design will list them explicitly and a
  follow-up may extend the list when B.7 finalises type implementations.
- §6.6 — `Policy.Action` accepted strings: `allow | log | mask | block`
  per `neuraltrust-domain.mdc`.
- §6.7 — Listings: offset pagination (`?page`, `?size`, default 1/20,
  max 200), substring `?name=` filter only.
- §6.8 — Repository layout: **subfolder per entity** (AGENT.md §11
  wins over TrustGate's flat shape).
- §6.9 — Tests: unit + handler tests in every PR;
  repository integration tests behind `PG_TEST_URL` (already documented
  in AGENT.md §9).
